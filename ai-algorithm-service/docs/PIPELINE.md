# Pipeline: Sim Bundle -> Real Runtime

Tài liệu này là bản đồ end-to-end của service: training tạo sim bundle, Core/backend sync topology thật, ai-ops compose runtime bundle, ai-runtime phục vụ inference.

## 1. Flow summary

```text
Training team
  build *.sim.zip
  upload MinIO/S3

Core/backend
  export real topology snapshot
  provide confirmed simToReal mapping
  PUT /internal/sync/areas/{area_id}/real-network

ai-ops
  compile real normalization
  pull sim bundle
  compose runtime bundle
  write compatibility_report.json
  activate bundle after review

ai-runtime
  reload active bundle
  serve POST /api/algorithm/ai
```

## 2. Artifacts

### 2.1 Sim bundle

Training bundle is a ZIP, usually ending with `.sim.zip`.

Required logical contents:

```text
sim_bundle_manifest.json
policy.onnx
policy_meta.json
sim_network.json
intersection_configs/
```

The manifest declares tenant, network, version, schema version, and file paths.

### 2.2 Real network snapshot

Sent by backend/Core management system:

```http
PUT /internal/sync/areas/{area_id}/real-network
X-Internal-API-Key: <key>
```

Required fields:

| Field | Source | Note |
|---|---|---|
| `sourceEventId` | backend event | Idempotency key |
| `tenantId` | backend config | Customer/tenant |
| `networkId` | backend config | Must match sim bundle target |
| `schemaVersion` | backend/service contract | Snapshot schema |
| `area` | DB management | Area row |
| `areaCrosses` | DB management | Area-cross rows |
| `crosses` | DB management | Cross rows; recommend `location: "lat,lon"` |
| `roads` | DB management | Road rows; recommend `coordinates` polyline |
| `cycles` | DB management | Cycle rows |
| `stages` | DB management | Stage rows |
| `simToReal` | operator/integration mapping | Not in DB; required for production |

DB management provides real topology only. `simToReal` is an overlay mapping:

```json
{
  "simToReal": {
    "33202549": 567001,
    "360082": 567002
  }
}
```

## 3. Real normalization

After snapshot sync, service compiles runtime static topology under local model storage:

```text
models/
  real_normalization/
    area_<area_id>/
      real_normalization.json
      network.json
      intersections/
        cross_<cross_id>.json
```

It contains:

- real crosses
- direction map
- road/static topology
- cycles/stages
- optional/explicit sim-to-real mapping

This directory is generated by AI Service from the latest stored real snapshot. It is not uploaded by Core Controller and should not be edited manually in production.

Verify before go-live:

```bash
curl -H "X-Internal-API-Key: $INTERNAL_API_KEY" \
  http://localhost:8002/internal/sync/areas/1/real-normalization
```

Every production cross should have a usable `direction_map`. If GPS is present, direction is inferred from geometry. Without GPS, legacy `from_cross_direction` / `to_cross_direction` is auto-detected as 4-dir or 8-dir cardinal encoding. Diagonal 8-dir codes are dropped instead of guessed.

## 4. Compose runtime bundle

When ai-ops receives a sim bundle, it:

1. Extracts and validates sim bundle.
2. Finds real snapshot by `tenantId/networkId`.
3. Builds `deployment_map.json`.
4. Validates compatibility.
5. Writes `compatibility_report.json`.
6. Creates runtime bundle.
7. Optionally activates it.

Production should set:

```text
SIM_BUNDLE_AUTO_ACTIVATE=false
```

Then manually review and activate.

## 5. Compatibility gates

| Gate | Meaning |
|---|---|
| `SIM_CROSS_NOT_MAPPED` | Sim cross has no valid real cross mapping |
| `REAL_CROSS_NOT_FOUND` | Mapping points to missing real cross |
| `STAGE_COUNT_MISMATCH` | Sim phase count does not match real stage count |
| `DIRECTION_MISSING_IN_REAL` | Real cross lacks road direction required by sim |
| `DEPLOYMENT_MAP_VALIDATION_EXCEPTION` | Deployment map validator failed |
| `AUTO_CROSS_MAPPING_BY_ORDER` | Warning only; production blocker |

Production rule:

- Errors block activation.
- `AUTO_CROSS_MAPPING_BY_ORDER` also blocks production activation.
- Fix by supplying explicit `simToReal` or confirmed `sim_tls_id`/`sim_cross_id`.

## 6. Runtime inference

Core Controller calls:

```http
POST /api/algorithm/ai
```

Runtime resolves active bundle by area/network, hydrates static topology, builds features, runs policy, applies guardrails, and returns signal plan.

Production inference payload is intentionally small. The service has already stored/compiled static topology from `real_network_snapshot`, so each request should only carry:

- `areaId`, `crossId`, `cycleId`.
- Current stage state: `stageId` plus `greenTime` or `duration`.
- Road traffic demand: `roadId`, speed, occupancy, queue, vehicle count/window, density if available.

Static fields such as road direction, saturation flow, lanes, speed design, cycle/stage metadata, yellow and red-clear are hydrated first from compiled real normalization. Active runtime bundle provides the policy/model and can contribute model-specific phase mapping. Legacy full payloads are still accepted for compatibility.

Core Controller must still:

- timeout 500 ms
- retry at most once
- validate response
- fallback fixed-time on any failure
- audit input/output by `X-Request-Id`

Schema: [../api_docs/run_ai_algorithm.md](../api_docs/run_ai_algorithm.md).

## 7. Race conditions

### Sim bundle arrives before real snapshot

Expected behavior:

- Bundle status becomes `pending_real_snapshot`.
- After real snapshot arrives, service retries compose.
- If compose succeeds, bundle becomes composed/active depending on activation config.

### Real topology changes

Core/backend must call:

```http
PUT /internal/sync/areas/{area_id}/real-network
```

Then verify real normalization and re-compose/review any affected runtime bundle.

| Data | Owner |
|---|---|
